# 投资基金自动化交易系统 — 建设方案

> 版本：v1.0  
> 日期：2026-05-20  
> 目标：基于 AI 的基金推荐、每日涨跌总结、实时加减仓/换基建议，并逐步形成可执行的自动化交易闭环。

---

## 一、需求拆解与边界

### 1.1 核心能力


| 能力     | 说明                        | 优先级 |
| ------ | ------------------------- | --- |
| 基金推荐   | AI 结合市场、风格、风险偏好推荐具体基金     | P0  |
| 每日涨跌总结 | 持仓与关注池的净值、收益率、对比基准        | P0  |
| 实时操作建议 | 加仓 / 减仓 / 持有 / 换基，含理由与置信度 | P0  |
| 自动化执行  | 按规则或确认后下单（申购/赎回/转换）       | P1  |
| 风控与审计  | 限额、止损、操作日志、可回溯            | P0  |


### 1.2 必须提前明确的边界（避免后期返工）

1. **交易市场**：仅 A 股公募基金，还是含 QDII、ETF、场外+场内？A股 场内
2. **自动化程度**：
  - **半自动**：AI 出建议 → 人工一键确认 → 调用渠道下单（推荐起步） 人工下单 
  - **全自动**：定时/事件触发直接下单（需券商/第三方 API、合规与风控更严）
3. **资金与账户**：单账户还是多账户；是否对接具体平台（支付宝、天天基金、券商 APP、银行）。单账户、支付宝
4. **投资风格**：定投为主、波段为主，还是资产配置再平衡。都可以
5. **免责声明**：本系统为**辅助决策工具**，不构成投资建议；实盘需自行承担风险并遵守当地法规。

### 1.3 国内现实约束（方案设计时必须考虑）

- **公募基金大多无面向个人的官方「全自动交易 API」**；常见路径是：  
  - 半自动：生成「今日操作清单」+  Deep Link / 复制参数，人工在 APP 完成； 以这个为主 
  - 半自动：对接**支持 Open API 的券商/第三方**（需签约、验资、沙箱）；  
  - 全自动：通常限于 **ETF 场内**（券商 QMT/Ptrade 等）或机构通道。
- **净值披露**：场外基金净值一般 **T 日收盘后 T+1 公布**，「实时」对场外基金实为**盘中估值（估算）+ 昨日净值**，需在文案中区分。  
- **申赎规则**：确认日、到账日、最低持有、7 天赎回费等，必须在规则引擎中硬编码。  
- **合规**：不得承诺收益；日志留存；若对外提供服务可能涉及**投资咨询/基金销售**资质问题。

## 二、你需要准备什么（清单总览）

### 2.1 业务与合规准备

- 确定目标用户（仅本人 / 家庭 / 是否对外）  
- 书面：**风险偏好问卷**（保守/稳健/积极、最大回撤、投资期限）  
- 书面：**投资策略说明书**（什么情况下加仓、减仓、换基、清仓）  
- 基金交易账户（1～2 个主渠道即可，避免多平台持仓分散）  
- 若全自动：确认券商是否支持 **QMT/Ptrade/OpenAPI** 及费用  
- 税务与记录：分红、赎回收益是否需要导出给报税/记账

### 2.2 数据准备

- **基金主数据**：代码、名称、类型、经理、规模、费率、业绩比较基准  
- **净值历史**：日净值、复权净值（用于回测与涨跌总结）  
- **盘中估值**（可选）：用于「 quasi 实时」展示  
- **指数与宏观**：沪深300、中证500、国债指数、利率、汇率（QDII）  
- **持仓与成交**：份额、成本、申购日、可用份额（含在途）  
- **关注池 / 候选池**：AI 只在池内推荐，防止幻觉推荐不存在的基金  
- **新闻与公告**（可选）：季报、经理变更、大额限购

**数据来源建议（按易用程度）：**


| 类型      | 可选来源                              | 备注              |
| ------- | --------------------------------- | --------------- |
| 净值/基本信息 | 天天基金、东方财富、Tushare Pro、聚宽、Wind（付费） | 注意协议与频率限制       |
| 指数行情    | 交易所、Tushare、AkShare               | 实时性要求高的用付费或券商行情 |
| 持仓      | 手动录入 / Excel 导入 / OCR 对账单         | 初期可半自动          |
| 研报摘要    | RSS、财经 API + LLM 摘要               | 注意版权            |


### 2.3 技术与基础设施准备

- 运行环境：本地 Windows / Linux 服务器 / 云主机（定时任务需常开）  
- 数据库：PostgreSQL 或 SQLite（个人起步 SQLite 即可）  
- 缓存/队列（可选）：Redis + 定时任务（Celery/APScheduler）  
- **LLM API**：OpenAI、Claude、DeepSeek、通义等（需 API Key 与月度预算）  
- 密钥管理：`.env` 不入库，用环境变量或系统密钥链  
- 备份：数据库每日备份、配置版本化（Git）  
- 监控：任务是否成功、数据是否更新、API 额度

### 2.4 AI 与策略准备

- **System Prompt 库**：角色（基金研究员）、输出 JSON Schema、禁止编造基金代码  
- **RAG 知识库**（推荐）：自家策略文档、历史总结、基金说明书摘要  
- **评估样本**：10～20 个历史交易日，人工标注「应加仓/减仓」，用于对比 AI 建议质量  
- **规则引擎（硬规则优先于 AI）**：  
  - 单基金仓位上限（如 20%）  
  - 行业/主题集中度上限  
  - 单日最大申购金额  
  - 止损线（如 -15% 强制减仓提醒）  
  - 7 日内赎回预警

### 2.5 运营与日常使用准备

- 每日固定流程时间（如 20:30 净值更新后跑总结，21:00 出操作建议）  
- 推送渠道：邮件 / 企业微信 / Telegram / 本地 Markdown 报告  
- 人工确认 SLA：全自动前至少保留 7～30 天「只建议不下单」观察期

### 3.1 模块说明


| 模块   | 职责                                    |
| ---- | ------------------------------------- |
| 数据采集 | 定时拉净值、指数；失败重试与告警                      |
| 特征计算 | 日涨跌、阶段收益、相对基准超额、最大回撤、波动率              |
| 每日总结 | 模板 + 数据填充 + LLM 润色（数据以程序计算为准，LLM 只解释） |
| 建议引擎 | 规则引擎 → 结构化上下文 → LLM → JSON 校验 → 风控否决  |
| 交易执行 | 生成订单意图；半自动导出；全自动调券商 API               |
| 审计   | 每次建议的输入快照、模型版本、输出、是否执行                |


**关键原则：数字由程序算，AI 负责归纳与建议；所有建议过 Schema 校验与基金白名单。**

## 四、核心业务流程

### 4.1 每日涨跌总结（建议交易日 20:30 后触发）

```
同步最新净值 → 计算持仓盈亏 → 对比基准与关注池 → 生成结构化数据
    → LLM 生成「人话总结」→ 推送/落盘 reports/YYYY-MM-DD.md
```

**总结内容模板：**

- 账户总市值、当日盈亏、累计盈亏  
- 各持仓：日涨跌、占仓位、距成本盈亏  
- 关注池 Top 变动  
- 相对基准（如沪深300）表现一句话  
- 明日待办（是否有定投、是否有开放申购限制）

### 4.2 实时加减仓 / 换基建议

**「实时」定义建议：**


| 品种     | 可用数据频率            | 建议刷新策略                         |
| ------ | ----------------- | ------------------------------ |
| 场外基金   | 估值 15min 级 + 净值日频 | 盘中 2～4 次扫描 + 收盘后 definitive 建议 |
| 场内 ETF | 行情秒级              | 可更高频，但需控制 API 与交易成本            |


**建议输出 JSON Schema（示例）：**

```json
{
  "date": "2026-05-20",
  "actions": [
    {
      "fund_code": "161725",
      "fund_name": "某某混合",
      "action": "reduce",
      "ratio": 0.1,
      "reason": "主题仓位超上限；近20日波动上升",
      "confidence": 0.72,
      "rule_hits": ["POSITION_CAP", "VOLATILITY"],
      "requires_human_confirm": true
    }
  ],
  "switch_candidates": [],
  "overall_risk_level": "medium"
}
```

**决策流水线：**

1. 硬规则扫描（超限、止损、限购、持有期）
2. 量化信号（均线、回撤、相对基准、可选宏观标签）
3. LLM 综合解读与生成 `reason`（不得改变 `action` 硬规则已否决的指令）
4. 人工确认（半自动）或风控二次校验（全自动）
5. 执行并写审计日志

### 4.3 基金推荐（换基 / 新配置）

- 维护 **候选基金池**（50～200 只），由你定期筛选，AI 仅在池内排序与解释。  
- 输入：策略标签、历史表现、费率、规模、经理任期、相关性（与现有持仓）。  
- 输出：推荐列表 + 替换谁 + 预期仓位 + 不适用情形。  
- **换基**需单独模块：赎回 A → 申购 B 的空窗期、双倍手续费、7 天限制。

## 六、风控与合规（不可省略）

1. **仓位与集中度**：单基、单行业、单主题上限。
2. **流动性**：小规模基金、大额限购、暂停申购。
3. **行为约束**：单日/单周最大交易额；全自动必须「*kill switch*」一键暂停。
4. **模型风险**：LLM 幻觉 → **基金代码白名单** + JSON Schema 强校验。
5. **审计**：`who / when / what / input_hash / model_version / output`。
6. **用户协议**：系统仅为辅助工具，历史回测不代表未来收益。

## 七、分阶段实施路线图

### Phase 0 — 准备（1 周）

- 确定策略说明书与基金候选池（Excel 即可）  
- 注册数据 API、LLM API  
- 手动维护第一版持仓表

### Phase 1 — 数据与日报（2～3 周）

- 净值自动同步 + 持仓导入  
- 每日涨跌 **纯程序** 报告（先不用 AI）  
- 验证数据准确性

### Phase 2 — AI 总结与建议（2～3 周）

- LLM 日总结润色  
- 规则引擎 + AI 结构化建议（**仅推送，不交易**）  
- 30 天纸面跟踪建议质量

### Phase 3 — 半自动执行（2～4 周）✅ 已实现

- 生成「操作清单」`data/checklist/`  
- QQ 邮件推送（`config/notify.yaml` + `.env` SMTP）  
- Windows 计划任务 `scripts/register_scheduled_task.ps1`  
- 小资金策略放宽、沪深300 多源拉取+缓存  

### Phase 4 — 全自动与进阶（可选）

- ETF 场内自动化、再平衡、回测框架、多策略 A/B  
- 强化监控、灾备、费率优化

## 附录 C：与 AI 协作的 Prompt 设计要点

1. **输入只给结构化 JSON**（净值、仓位、规则触发项），不给长篇原始行情。
2. **输出必须 JSON Schema**，便于程序解析与拒绝非法基金代码。
3. **明确约束**：「仅可从 fund_universe 中选择」「不得保证收益」「不确定时 action=hold」。
4. **版本化 Prompt 文件**：`prompts/daily_advice_v1.txt`，变更可回溯。
